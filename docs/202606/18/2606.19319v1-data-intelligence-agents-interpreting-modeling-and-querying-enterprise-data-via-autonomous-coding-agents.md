## 摘要
生产数据集成面临的主要瓶颈是数据所有者、工程师和分析师之间反复、有损的交接，他们必须协作发现、组织和查询企业数据。我们提出数据智能代理（DIA），这是一个由三个代理（数据解释器、模式创建器和查询生成器）组成的系统，通过将自主编码代理（ACA）视为一级抽象来压缩这一工作流：代理不是生成文本，而是生成、执行、验证和修复具体工件，利用共享内存实现经验复用，并将每个工件展示给领域专家审查。DIA已在企业客户的生产环境中部署。我们深入研究查询生成器，并在完全自主模式下，跨七个SQL基准测试（涵盖四个任务类别和四种方言）对其进行评估。它在所有七个基准测试中均达到或超越了最佳已发表结果，表明这种基于执行、构建在ACA和共享内存之上的架构能够泛化到数据智能工作负载，仅通过自然语言指令进行适配。

## Abstract
Production data integration is bottlenecked by repeated, lossy handoffs between data owners, engineers, and analysts who must collaboratively discover, structure, and query enterprise data. We present Data Intelligence Agents (DIA), a system of three agents (Data Interpreter, Schema Creator, and Query Generator) that compresses this workflow by treating autonomous coding agents (ACAs) as a first-class abstraction: rather than emitting text, the agents generate, execute, validate, and repair concrete artifacts, draw on a shared memory for experience reuse, and surface each for review by domain experts. DIA is deployed in production for enterprise customers. We study the Query Generator in depth and evaluate it in fully autonomous mode across seven SQL benchmarks spanning four task categories and four dialects. It matches or surpasses the best published results on all seven, demonstrating that an architecture grounded in execution, built on ACAs and a shared memory, generalizes across the data intelligence workload with adaptation confined to natural-language instructions.

## 1. 论文的核心问题与整体含义

- **研究动机**：企业数据集成流程中，数据所有者、工程师、分析师之间存在多次、有损的交接，每次交接都会丢失上下文并增加延迟。领域专家理解数据含义，但需要工程师来结构化和验证，再由分析师查询，这一串联流程效率低下。
- **关键洞察**：大型语言模型使单个步骤看似可行，但现有系统只覆盖工作流的片段，未能形成一个闭环。本文主张将自主编码代理（Autonomous Coding Agent, ACA）作为一级抽象，通过生成、执行、验证和修复可执行工件来压缩整个流程，并让领域专家审查每个工件。

## 2. 论文提出的方法论

- **核心思想**：构建一个由三个代理组成的系统（Data Interpreter, Schema Creator, Query Generator），它们共享一个工作空间（shared workspace）和一个共享记忆（shared memory），并由同一个ACA驱动。ACA在沙箱环境中编写并执行代码，所有输出均为可执行、可审查的工件。
- **关键技术细节**：
  - **共享记忆**：基于工件，分为三个层级：检索示例（episodic examples）、会话经验（session lessons）、跨会话经验（cross-session lessons）。记忆是拉取式的，且在使用前会通过实时数据探针验证其前提条件。
  - **Data Interpreter**：对原始异构数据（CSV/JSON/Excel）编写并执行分析代码，生成结构化解释 $P$，包含推断的schema、列分布、空值统计、候选键和连接路径。
  - **Schema Creator**：基于解释$P$和原始数据，生成并执行加载和验证代码，构建物理关系数据库 $\beta$，包括表、键约束和完整性约束，并执行四项验证（行数核对、列覆盖、键有效性、加载完整性）。
  - **Query Generator**：针对自然语言问题 $q$，生成并执行SQL。流程包括：
    1. **形状声明**：从问题中推导预期结果形状 $\kappa$（列、粒度、排序、过滤器）。
    2. **Schema探索**：执行轻量级探针查询确认键、列值和基数假设。
    3. **生成与执行**：生成候选查询 $y = G(q, \Sigma, M^*, \kappa)$ 并执行得到结果 $R$。
    4. **自验证**：检查结果 $R$ 是否符合 $\kappa$，若不符合则诊断、修改并重新执行，直到通过或达到预算。

- 共享记忆机制能够有效积累跨数据库的经验规则（如使用COUNT(DISTINCT pk)避免多表连接中的重复计数），这些规则是人工可读的，且在使用前会通过数据探测验证。

## 7. 优点

- **统一抽象**：将ACA而非LLM作为核心抽象，所有输出均为可执行、可审查的工件，替代了传统文本交接。
- **执行接地**：通过实际执行来验证结果，而非仅依赖LLM输出的文本逻辑，提高了可靠性。
- **共享记忆**：三个代理共享同一记忆，跨任务复用经验，且采用拉取+验证机制，避免过时规则被错误使用。
- **任务泛化**：单一系统覆盖了数据智能工作流中的多个环节（解释、建模、查询），无需为每个子任务训练独立模型或设计专用pipeline。
- **生产部署**：已在企业客户生产环境中使用（附录G），证明了实际可行性。

## 8. 不足与局限

- **计算效率**：每个问题需经过生成-执行-验证的迭代循环，对话任务还需多轮交互，平均耗时1~10分钟，不适合高吞吐或实时交互场景。论文未报告详细的token/成本分析。
- **验证不具语义**：自验证仅检查结果形状（列、行数、排序等），不检查内容语义，因此当代理误解问题意图时，验证可能同样错误地通过。
- **评估范围局限**：
  - 仅详细评估了Query Generator，Data Interpreter和Schema Creator的能力未进行独立基准测试。
  - 仅使用单一LLM（Claude Sonnet 4.5），未测试对其他LLM的敏感性。
  - 对话任务使用了模拟用户而非真实用户，可能高估了实际交互效果。
  - 共享记忆仅进行了定性分析，缺乏量化消融实验以评估其贡献度。
- **失败主要集中在推理错误**：特别是当问题表述不精确时，代理可能错误理解连接路径、过滤条件或计算公式，这是当前LLM能力边界所致。

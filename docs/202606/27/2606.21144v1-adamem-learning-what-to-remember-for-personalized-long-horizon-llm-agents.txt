Abstract 

Long-term memory systems for Large Lan-guage Model (LLM) agents typically try to 

remember everything , extracting memories uni-formly to retain as many facts as possible. In production, however, inference cost and finite context budgets make this untenable: beyond consolidating raw dialogue into memory, an agent must exert write control , efficiently keep-ing only the information each user actually cares about. Otherwise, long-horizon person-alized interactions suffer memory bloat , where irrelevant trivia crowds out useful informa-tion and steadily erodes question-answering (QA) accuracy. We argue that what is worth remembering is role-dependent, and propose 

AdaMem (Adaptive Memory), a method that 

learns what to remember for each user from feedback. AdaMem maintains a structured, role-specific Memory Policy and refines it from weekly QA feedback through a lightweight, patch-style self-reflection step with failure roll-back. To study this setting, we build AdaMem-Bench , a benchmark that simulates weeks of interaction with week-by-week QA. Across two extraction models and two feedback modes, AdaMem improves QA accuracy by up to 

+9.0% over the uniform Mem0 baseline while shrinking memory volume by 9% .

1 Introduction 

Large Language Model (LLM) agents are increas-ingly deployed in long-horizon personalized scenar-ios, where consistent long-term memory underpins trust and personalization (Park et al., 2023; Wang et al., 2024). To overcome context-window lim-its, existing memory systems (e.g., Mem0, Mem-oryBank) adopt a Retrieval-Augmented Genera-tion (RAG) architecture (Lewis et al., 2020; Packer et al., 2023; Zhong et al., 2024) and operate under a uniform extraction rule : they try to remember everything, saving every factual statement on the

In real-world de-ployment, this collides with two hard constraints— inference cost and a finite context budget—since every memory that is later retrieved consumes to-kens and latency. Consolidating raw dialogue into memory is therefore not enough: an agent must also control what it writes , retaining only the infor-mation each user cares about. As interactions span weeks or months, this uni-formity backfires, producing memory bloat : the memory bank accumulates irrelevant trivia (e.g., casual greetings, minor scheduling details) that crowds out the memories that matter at retrieval time, steadily degrading QA accuracy (Figure 1). Our key insight is that an agent should not re-member everything, but instead learn what to re-member , because what counts as important is sub-jective and role-dependent: an agent talking to a manager should track business decisions, whereas one chatting with a fitness buddy should track work-out schedules. Memory extraction should therefore be guided by a Memory Policy aligned with each user’s preferences and continually corrected from 1

(2) We propose AdaMem , which learns a role-specific Memory Policy from feedback and applies patch-style reflection with failure rollback to decide what to write—a lightweight recipe that drops into exist-ing RAG memory stacks. 2 Related Work 

Long-Term Memory and Hierarchical Storage 

Long-term memory is essential for LLM agents in long-horizon interactions (Wang et al., 2024). Early works retrieve raw chat history (Park et al., 2023) or extract atomic facts (Mem0, 2024; Zhong et al., 2024). To handle context limitations, MemGPT (Packer et al., 2023) and MemoryOS (for Com-putational Linguistics 2025 et al., 2025a) use hi-erarchical storage paging. Graph-based architec-tures like MemInsight (for Computational Linguis-tics 2025 et al., 2025b) and All-Mem (Lv et al., 2026) build topological relations to prevent struc-ture drift, while systems like Temporal Semantic

Memory (TSM) (Su et al., 2026) organize events along semantic timelines to resolve temporal rea-soning. 

Personalization, Preference Alignment, and Al-leviating Bloat To personalize agent behavior (Hao et al., 2025), systems like PersonaAgent (Zhang et al., 2025), PersonaMem (Zhang et al., 2026), and Dynamic Affective Memory (Lu and Li, 2025) align agent actions or emotions using retrieved long-term memory. Yet they still remem-ber everything at extraction time, so the underlying store suffers from memory bloat. SimpleMem (Liu et al., 2026a) applies semantic compression and Nemori (Ma et al., 2025) uses predictive-coding theory to trigger storage on prediction gaps, but both compress after the fact. AdaMem instead de-cides what to remember upfront, via a structured, role-specific Memory Policy that guides extraction. 

Active Memory Policies and Self-Reflective Loops Memory management has shifted from static heuristics to active policies. AgeMem (Yu et al., 2026), for instance, frames CRUD opera-tions as an action space trained via GRPO. For self-correction, dialogue systems adopt reflective memory management (RMM) (Tan et al., 2025) for prospection and retrospection, while EvolveMem (Liu et al., 2026b) learns retrieval configurations from failure logs. Unlike task- or code-level re-flection (Shinn et al., 2023; Madaan et al., 2023), AdaMem applies patch-style reflection to the ex-traction policy itself, iteratively refining the natural-language rules that decide what to remember for each role—and, in doing so, exposes policy infer-ence as the key bottleneck.

At the end of each week, the agent reflects on in-correct QA pairs Wt and diagnoses failures as: (1) 

Recall Failure (Missing Memory) where the dia-logue had the info but the extractor missed it (pol-icy too restrictive); or (2) Precision Failure (Noisy Memory) where irrelevant retrieved memories mis-led the answering model (policy too permissive). Based on this, the agent generates a policy patch 

∆Pt to adjust Ti. To ensure stability and prevent policy drift, AdaMem employs a failure rollback 

mechanism: if the updated policy degrades valida-tion performance on historical QA pairs, the update is rejected and the policy rolls back. 

3.3 Rendering and Time Anchoring 

During daily extraction, the active policy P is ren-dered as custom instructions (e.g., “ Extract mem-ories strictly according to the following character preferences... ”) in the extractor’s prompt. 

Time Anchoring Relative temporal references (e.g., “tomorrow”) become misleading if saved ver-batim. AdaMem injects the session’s absolute date into the extraction prompt and instructs the LLM to resolve all relative expressions into absolute dates before saving (e.g., “Meeting scheduled for June 18, 2026”). 

4 AdaMem-Bench and Setup 

We construct AdaMem-Bench , a personalized long-horizon benchmark with session-level golden-memory annotations, to evaluate the AdaMem method. 

4.1 Dataset and Data Synthesis 

To evaluate preference-guided memory systems under realistic settings, we build AdaMem-Bench around three design principles: • Realistic conversational dynamics. We sim-ulate natural, multi-session dialogues that em-bed casual chit-chat, temporal references, and subjective nuances, rather than single-turn or template-based text. • Multi-persona memory tracking. Each envi-ronment involves concurrent interactions with several characters, each with distinct roles and information preferences, so the agent must track persona-specific memory and retrieve accordingly. • Homogeneous concept interference. Dia-logues are designed with high semantic over-lap across characters (e.g., several people dis-cussing fitness schedules), and these same-category distractors stress whether the system confuses similar but irrelevant content from other personas. Our data synthesis pipeline consists of four main stages: 

Scenario and Persona Initialization We initial-ize N long-term interactive stories, each involving an assistant interacting with C characters. Each character is assigned a distinct preference profile (e.g., schedules, emotional states, deadlines) repre-senting their ground-truth preferences. 

Multi-Session Dialogue Generation We simu-late daily dialogues across T weeks using a multi-agent pipeline, naturally embedding key events, factual information, and casual chit-chat. 3Golden Memory Annotation Each dialogue is annotated with session-level “golden memories” (atomic facts matching characters’ preferences), treating other details as noise. 

Preference-Aligned QA Generation At each week’s end, we generate QA pairs targeting only preference-aligned information to test the agent’s long-term recall. 

4.2 Baselines and Temporal Alignment 

We compare AdaMem against three baseline meth-ods processed in the same chronological order: (1) 

Full Context : A theoretical upper bound where the entire raw history is fed directly into the LLM context. (2) Ideal Memory : A bound where only human-annotated “golden memories” are stored. (3) Mem0 : A widely used uniform extraction base-line (Mem0, 2024) that remembers everything, ex-tracting all factual information without preference guidance. QA answers are always produced by 

deepseek-v4-flash ; the two models we vary (DeepSeek-V4-Flash and Gemini-3.5-Flash) are the extraction models used by Mem0 and AdaMem. Because Full Context and Ideal Memory involve no memory extraction or policy update, they are extractor-independent and reported once rather than per model. 

4.3 Feedback Modes 

We evaluate two feedback modes after incorrect QA answers: (1) Explicit Feedback : The user provides the correct answer and explains their pref-erence (e.g., “ Incorrect. You should remember this because I only care about schedules ”). (2) Im-plicit Feedback : The user only provides the cor-rect answer, and the agent must implicitly infer the underlying preference. 

4.4 Evaluation Metrics 

We employ a decoupled evaluation methodology: 

QA Accuracy The percentage of weekly QA questions answered correctly, evaluated by a de-coupled deepseek-v4-flash judge. 

Extraction F1 The harmonic mean of Precision (the proportion of extracted memories matching user preferences) and Recall (the proportion of suc-cessfully extracted golden memories). Note that precision here is preference-based, not factual. 

Memory Efficiency Ratio (MER) Measures the trade-off between QA accuracy and memory vol-ume, defined as: 

MER = Acc /MemVol Acc Ideal /MemVol Ideal 

(1) normalized against Ideal Memory ( MER Ideal =1.0), showing how efficiently each memory con-tributes to accuracy. 

5 Results and Analysis 

We evaluate AdaMem to answer four research ques-tions: • RQ1 (Overall Performance) : Does AdaMem outperform the uniform memory baseline in terms of QA accuracy and memory compact-ness across different models and feedback modes? • RQ2 (Information Categories) : Where do the performance gains of AdaMem come from when analyzed across different categories of user preferences? • RQ3 (Retrieval Dynamics) : How does AdaMem affect retrieval quality and mitigate memory bloat over extended multi-week inter-action horizons? • RQ4 (Policy Learning) : What is the primary bottleneck in preference-guided memory: pol-icy adherence (following instructions) or pol-icy inference (learning preferences from feed-back)?

5.2 RQ2: Granular Performance across Information Categories 

To understand why AdaMem achieves these gains, we break down the accuracy delta between AdaMem and Mem0 by the information category of the questions (Table 3). The results reveal a clear pattern: • For Fact/Number , both Mem0 and AdaMem reach near-perfect accuracy ( ≥ 93% ), with AdaMem providing a marginal +1.1% gain, since numbers and concrete facts are highly salient and easily captured by any standard extractor. • For Emotion/Attitude , Schedule/Time , and 

Decision/Conclusion , AdaMem achieves larger gains of +15.9% , +9.9% , and +9.0% ,respectively. These categories represent “soft preferences”—information that is contextual and subjective. A uniform extractor often discards emotional states or minor schedule adjustments as noise, whereas AdaMem’s preference-guided policy explicitly instructs the model to retain them.

5.4 RQ4: Policy Adherence vs. Policy Inference 

To identify the core bottleneck, we decouple the agent’s ability to follow a policy from its ability to 

infer the correct one. 

Policy-Following Rate Using an independent judge to check whether each extracted memory matches the active policy, both models score highly: DeepSeek 81.1% /80.8% (Explicit/Implicit) and Gemini 86.1% /84.8% . Thus instruction adher-ence is not the primary bottleneck : modern LLMs readily follow structured meta-instructions. 

Policy-Convergence (Alignment with Ground-Truth) We measure the semantic alignment of the learned policies with ground-truth preferences over the 10-week horizon (Figure 2b). The curves reveal a clear contrast: • Under Explicit Feedback , both models steadily converge toward ground-truth, reach-ing 88.3% and 81.7% alignment by Week 10. • Under Implicit Feedback (correctness-only signals), policy learning stalls, flatlining be-tween 53.3% and 63.3% alignment. This explains the Explicit–Implicit accuracy gap in Table 2 (e.g., DeepSeek: 85.2% vs. 82.0%) and exposes a limitation of current LLMs: they re-liably follow explicit instructions but struggle to reverse-engineer preferences from correct-ness feedback alone , making policy inference un-der weak feedback a key frontier for personalized agents. 

6 Conclusion 

We introduced AdaMem, a preference-guided framework that shifts personalized long-horizon LLM agents from remembering everything to 

learning what to remember . By refining a role-specific Memory Policy from weekly QA feed-back, AdaMem exercises write control over mem-ory, mitigating bloat and retrieval noise over long timelines. This carries direct industrial value: un-der production inference-cost and context budgets, keeping memory compact and user-aligned lowers retrieval and inference overhead while improving accuracy, and the policy patches drop into exist-ing RAG memory stacks without retraining. On 6our AdaMem-Bench benchmark, AdaMem consis-tently beats uniform baselines in both QA accuracy and memory efficiency across two models and two feedback modes. Finally, our analysis shows that while instruction adherence is reliable in modern LLMs, inferring preferences from implicit feed-back remains a key bottleneck and an important future direction. 

7 Limitations 

We acknowledge several limitations: 1. Synthetic data dependency. Our evaluation is conducted on a synthesized long-horizon dataset. Although it is highly controlled and annotated with precise golden memories, syn-thetic dialogues may not fully capture the lin-guistic diversity, conversational noise, and er-ratic preference shifts of real-world human interactions. 2. Reliance on LLM-as-a-judge. Our eval-uation of QA correctness, extraction qual-ity, policy following, and policy align-ment relies on LLM-as-a-judge methods. Although we decoupled the judge model (deepseek-v4-flash ) from the answering models and used strict schemas, LLM judges can still exhibit subtle biases. 3. Evaluation scale. Due to API cost constraints, our evaluation is limited to 5 stories and 10 weeks of interaction. A larger cohort of sto-ries and longer horizons (e.g., full-year in-teractions) would provide stronger statistical guarantees. 4. Policy drift and over-fitting. Under implicit feedback, the patch-style reflection mecha-nism can occasionally over-fit to a single QA failure, producing an overly restrictive patch that degrades other categories. Our rollback mechanism mitigates this, but stronger policy regularization is needed.
